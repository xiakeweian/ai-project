# Feature Specification: 个人博客系统

**Feature Branch**: `001-personal-blog-system`  
**Created**: 2026-04-11  
**Status**: Draft  
**Input**: User description: "构建一个功能完整的个人博客系统，包含用户系统、文章系统、标签与分类、前后台页面、ISR 优化与图片上传能力。"

## Clarifications

### Session 2026-04-11

- Q: 新注册用户的默认角色是什么，以及谁可以授予更高角色？ → A: 所有新注册用户默认是 `reader`，由管理员分配 `author` 或 `admin`。
- Q: 当 GitHub OAuth 返回的邮箱已存在于邮箱密码账号中时，系统应如何处理？ → A: 用户必须先登录原邮箱账号，再手动绑定 GitHub 身份，不允许自动合并账户。
- Q: 已登录读者提交评论后，评论应立即公开还是先审核？ → A: 评论默认立即公开，管理员可事后隐藏或删除。
- Q: 文章详情 URL 应使用什么标识，以及是否允许修改？ → A: 使用站点内唯一的可读 `slug`，自动从标题生成，作者可在发布前修改；发布后保持稳定。
- Q: 删除仍有关联文章的分类时，系统应如何处理？ → A: 管理员必须先选择替代分类，并将相关文章迁移后才能删除原分类。

## User Scenarios & Testing（用户场景与测试）

### User Story 1 - 公共阅读与内容发现 (Priority: P1)

作为未登录访客或普通读者，我希望能够浏览博客首页、搜索文章、按标签和分类筛选，并进入文章详情页阅读内容，这样我可以快速找到并消费感兴趣的内容。

**Why this priority**: 这是博客系统对外提供价值的核心能力。即使后台与复杂管理能力尚未完成，只要访客能够稳定发现并阅读已发布文章，系统就已经具备对外可用的 MVP 价值。

**Independent Test**: 准备若干篇已发布文章、多个标签和分类后，可以独立验证首页分页、关键词搜索、标签页筛选、分类页筛选、文章详情渲染、关于页访问以及阅读量展示是否工作正常。

**Acceptance Scenarios**:

1. **Given** 系统中存在多篇已发布文章，**When** 访客进入首页，**Then** 系统应按时间倒序展示最新文章列表，并支持分页切换。
2. **Given** 某篇文章属于指定标签和分类，**When** 访客通过标签页或分类页筛选内容，**Then** 系统应仅展示匹配条件的已发布文章。
3. **Given** 某篇文章内容为 Markdown，且存在唯一 `slug`，**When** 访客通过文章详情 URL 进入页面，**Then** 系统应正确渲染 Markdown 内容，并显示作者信息、发布时间、标签与阅读量。
4. **Given** 访客输入关键词，**When** 执行搜索，**Then** 系统应返回标题、摘要或内容中匹配关键词的已发布文章列表。
5. **Given** 访客访问关于页，**When** 页面加载完成，**Then** 系统应展示预设的静态关于内容。

---

### User Story 2 - 用户认证与角色访问控制 (Priority: P1)

作为平台用户，我希望可以通过 GitHub OAuth 或邮箱密码完成注册和登录，并根据自己的角色看到正确的权限范围，这样系统既易于使用，也能保护后台与内容安全。

**Why this priority**: 博客系统包含作者与管理员后台能力，必须先建立清晰的身份认证和角色边界，才能安全地承载内容创建、后台管理和评论行为。

**Independent Test**: 分别创建管理员、作者、读者账号，使用邮箱密码登录与 GitHub OAuth 登录流程，验证登录成功、会话建立以及不同角色访问不同页面和操作时的权限差异。

**Acceptance Scenarios**:

1. **Given** 新用户尚未注册，**When** 用户通过邮箱和密码完成注册，**Then** 系统应创建默认角色为 `reader` 的用户账户并允许其登录。
2. **Given** 用户已配置可用的 GitHub OAuth，**When** 用户选择 GitHub 登录，**Then** 系统应完成 OAuth 登录并创建或绑定对应账户；若返回邮箱已存在于邮箱密码账号中，则必须要求用户先登录原账号后再手动绑定 GitHub 身份。
3. **Given** 当前用户角色为读者，**When** 其访问后台文章管理或用户管理页面，**Then** 系统应拒绝访问并给出明确的权限提示。
4. **Given** 当前用户角色为作者，**When** 其尝试访问用户管理模块，**Then** 系统应拒绝访问。
5. **Given** 当前用户角色为管理员，**When** 其访问管理后台，**Then** 系统应允许其查看和管理所有内容与用户。

---

### User Story 3 - 作者写作与发布工作流 (Priority: P2)

作为作者，我希望能够新建、编辑、删除和管理自己的文章，并在编写 Markdown 内容时实时预览效果，同时维护封面图、标签、分类、摘要和 SEO 信息，这样我可以完整地完成内容生产和发布。

**Why this priority**: 公共阅读能力建立后，内容持续供给是系统长期运转的关键。作者写作体验、状态流转和元数据管理决定了博客内容是否能稳定产出并高质量发布。

**Independent Test**: 使用作者账号独立验证文章新建、保存草稿、编辑、删除、发布、归档、Markdown 实时预览、封面图上传、SEO 信息填写，以及“仅能管理自己的文章”这一权限边界。

**Acceptance Scenarios**:

1. **Given** 作者已登录后台，**When** 新建文章并输入 Markdown 内容，**Then** 编辑器应实时展示预览结果。
2. **Given** 作者填写了标题、内容、摘要、分类和标签，**When** 保存文章为草稿，**Then** 系统应持久化文章并在文章管理列表中显示为草稿状态。
3. **Given** 作者拥有一篇草稿文章，**When** 将状态切换为已发布，**Then** 该文章应出现在前台可访问的文章列表和详情页中。
4. **Given** 作者上传封面图并配置 SEO 元信息，**When** 保存文章，**Then** 系统应保存封面地址以及 SEO 的 `title`、`description`、`og:image` 信息。
5. **Given** 作者尝试编辑或删除其他作者的文章，**When** 提交操作，**Then** 系统应拒绝该操作。

---

### User Story 4 - 管理员治理与内容后台 (Priority: P2)

作为管理员，我希望能够查看 Dashboard 统计数据，管理所有文章、分类、标签和用户，以便维护站点内容质量、信息结构和用户权限体系。

**Why this priority**: 当系统进入真实运营阶段后，必须具备统一治理能力，才能管理多角色内容生产、控制信息架构并维持系统秩序。

**Independent Test**: 使用管理员账号登录后台，独立验证 Dashboard 展示、文章全量管理、分类 CRUD、标签 CRUD、在用标签不可删除、用户列表与角色管理等关键能力。

**Acceptance Scenarios**:

1. **Given** 管理员已进入后台，**When** 打开 Dashboard，**Then** 系统应展示文章总数、按状态统计、总阅读量等核心指标。
2. **Given** 管理员查看文章管理列表，**When** 执行编辑、删除或状态切换，**Then** 系统应允许其操作任意文章。
3. **Given** 管理员创建、编辑或删除分类，**When** 提交操作，**Then** 系统应正确更新分类数据，并反映到文章筛选能力中；若删除分类时仍有关联文章，则必须先选择替代分类并完成迁移。
4. **Given** 某个标签已被一篇或多篇文章使用，**When** 管理员尝试删除该标签，**Then** 系统应阻止删除并提示标签仍在使用中。
5. **Given** 管理员查看用户列表，**When** 调整用户角色或账户状态，**Then** 系统应保存变更并立即影响权限判定。

---

### User Story 5 - 读者评论与互动 (Priority: P3)

作为读者，我希望在登录后可以对已发布文章发表评论，从而参与内容讨论；作为管理员，我希望能够在必要时处理不合规评论。

**Why this priority**: 评论能力能提升博客互动性，但它依赖于前台阅读和用户认证能力，因此优先级低于内容浏览和发布。

**Independent Test**: 使用读者账号进入已发布文章详情页发表评论，验证评论展示；再使用管理员账号验证评论删除或隐藏能力；同时验证未登录用户无法发表评论。

**Acceptance Scenarios**:

1. **Given** 读者已登录且正在查看已发布文章，**When** 提交评论内容，**Then** 系统应保存评论并默认立即在文章详情页展示。
2. **Given** 用户未登录，**When** 尝试发表评论，**Then** 系统应要求用户先登录。
3. **Given** 管理员发现违规评论，**When** 执行删除或隐藏操作，**Then** 系统应使该评论不再对前台读者可见。

### Edge Cases（边界场景）

- 当 GitHub OAuth 返回的邮箱已存在于邮箱密码账号中时，系统必须禁止自动合并账户，并要求用户先登录原邮箱账号后再手动绑定 GitHub 身份。
- 当作者尝试发布缺少必填字段的文章时，系统应阻止发布并指出缺失项。
- 当文章状态从 `published` 变为 `archived` 时，系统应将其从首页、搜索结果、标签页和分类页中移除，并默认禁止公众继续访问。
- 当标签正在被文章使用时，系统必须禁止删除该标签。
- 当分类被删除前仍有关联文章时，系统必须要求管理员先选择替代分类，并完成相关文章迁移后才能删除。
- 当读者在未登录状态下尝试评论时，系统应将其引导至登录流程，而不是静默失败。
- 当评论已公开后被管理员隐藏或删除时，前台读者不应再看到该评论。
- 当作者在文章发布前修改 `slug` 时，系统必须确保 `slug` 仍保持全站唯一；文章发布后不得再随意改变详情 URL 标识。
- 当文章详情被频繁刷新时，阅读量统计应避免在极短时间内被异常放大。
- 当图片上传失败或存储不可用时，系统应保留编辑中的文章内容并允许用户重试上传。
- 当搜索、标签或分类筛选无结果时，系统应展示明确的空状态，而不是空白页面。

## Requirements（需求）

### Functional Requirements

- **FR-001**: 系统必须支持用户通过邮箱和密码完成注册。
- **FR-002**: 系统必须支持用户通过邮箱和密码完成登录。
- **FR-003**: 系统必须支持用户通过 GitHub OAuth 完成登录。
- **FR-003a**: 当 GitHub OAuth 返回的邮箱已存在于邮箱密码账号中时，系统必须要求用户先登录原邮箱账号，再手动绑定 GitHub 身份；系统不得仅凭邮箱匹配自动合并账户。
- **FR-004**: 系统必须支持三种角色：管理员（admin）、作者（author）、读者（reader）；所有新注册用户默认角色为 `reader`，只有管理员可以授予或提升为 `author` 或 `admin`。
- **FR-005**: 系统必须基于用户角色执行访问控制，并阻止未授权用户访问后台页面或执行受限操作。
- **FR-006**: 管理员必须能够管理所有内容和所有用户。
- **FR-007**: 作者必须能够创建、编辑、删除和管理自己创建的文章，但不能管理其他作者的文章。
- **FR-008**: 读者必须能够浏览已发布文章，并对已发布文章发表评论。
- **FR-009**: 系统必须支持文章以 Markdown 格式编写，并在编辑过程中提供实时预览。
- **FR-010**: 系统必须为文章保存以下字段：标题、内容、摘要、封面图、标签、分类、状态和唯一 `slug`。
- **FR-011**: 系统必须支持文章状态 `draft`、`published`、`archived`，并允许具备权限的用户进行状态切换。
- **FR-012**: 只有 `published` 状态的文章可以在前台公开展示。
- **FR-012a**: 系统必须基于文章唯一 `slug` 提供公开详情页 URL；`slug` 应自动从标题生成，作者仅可在文章首次发布前修改，发布后应保持稳定。
- **FR-013**: 系统必须支持按关键词搜索文章，并支持按标签、分类筛选文章。
- **FR-014**: 系统必须支持文章同时绑定多个标签。
- **FR-015**: 系统必须要求文章归属且仅归属一个分类。
- **FR-016**: 文章详情页必须展示 Markdown 渲染后的正文、作者信息、发布时间、标签和阅读量。
- **FR-017**: 系统必须在文章详情页统计并展示阅读量。
- **FR-018**: 系统必须支持为文章配置 SEO 元信息，包括 `title`、`description` 和 `og:image`。
- **FR-019**: 首页必须展示最新已发布文章列表，并支持分页。
- **FR-020**: 系统必须提供标签页，并展示指定标签下的已发布文章。
- **FR-021**: 系统必须提供分类页，并展示指定分类下的已发布文章。
- **FR-022**: 系统必须提供一个静态的关于页。
- **FR-023**: 后台 Dashboard 必须展示至少以下统计数据：文章总数、各状态文章数量、总阅读量。
- **FR-024**: 后台文章管理必须支持文章列表查看、新建、编辑、删除和状态切换。
- **FR-025**: 系统必须支持分类的新增、查看、编辑和删除。
- **FR-025a**: 当分类仍有关联文章时，系统必须要求管理员先选择替代分类并迁移相关文章，再允许删除原分类。
- **FR-026**: 系统必须支持标签的新增、查看、编辑和删除。
- **FR-027**: 当标签仍被任一文章使用时，系统必须禁止删除该标签。
- **FR-028**: 管理员必须能够查看用户列表，并管理用户角色和账户状态。
- **FR-029**: 系统必须允许已登录读者在已发布文章下创建评论，且评论默认立即公开展示。
- **FR-030**: 管理员必须能够删除或隐藏任意评论；被隐藏或删除的评论不得继续对前台读者可见。
- **FR-031**: 系统必须支持文章封面图和正文图片上传。
- **FR-032**: 图片上传必须先支持本地存储，并预留云存储适配接口以支持后续扩展。
- **FR-033**: 首页和文章详情页必须支持 ISR 或等效的增量静态再生成机制，以提升访问性能。
- **FR-034**: 系统必须在移动端和桌面端都提供可用的响应式布局。
- **FR-035**: 系统必须在公开页面输出正确的 SEO 信息，以便搜索引擎和社交分享正确识别文章内容。
- **FR-036**: 系统必须为首页、搜索结果页、标签页和分类页提供无结果时的空状态反馈。

### Key Entities（关键实体）

- **User（用户）**: 表示系统登录主体，包含基础身份信息、角色、账户状态、登录方式和创建时间。
- **AuthIdentity（认证身份）**: 表示用户与认证提供方的绑定关系，可对应邮箱密码身份或 GitHub OAuth 身份。
- **Article（文章）**: 表示博客核心内容，包含标题、唯一 `slug`、摘要、Markdown 内容、封面图、状态、SEO 信息、发布时间、阅读量、作者和分类。
- **Tag（标签）**: 表示文章的主题标记，可与多篇文章建立多对多关系，并记录使用状态。
- **Category（分类）**: 表示文章所属栏目，一篇文章必须归属一个分类，一个分类可包含多篇文章；当分类被删除时，其关联文章必须先迁移到替代分类。
- **Comment（评论）**: 表示读者对已发布文章发表的内容，关联文章、评论作者、评论状态（至少支持可见、隐藏、删除）和创建时间。
- **MediaAsset（媒体资源）**: 表示上传的图片资源，包含存储位置、文件类型、大小、上传者和存储提供方信息。

## Success Criteria（成功标准）

### Measurable Outcomes

- **SC-001**: 未登录访客能够在不超过 3 次页面交互内，从首页、标签页、分类页或搜索结果进入任一已发布文章详情页。
- **SC-002**: 已登录作者能够在 5 分钟内完成一篇文章的“创建草稿、实时预览、填写元数据并发布”完整流程。
- **SC-003**: 在权限验收测试中，100% 的未授权操作都必须被阻止，包括读者访问后台、作者编辑他人文章、非管理员管理用户等场景。
- **SC-004**: 新发布或更新的首页和文章详情内容，必须在不超过 5 分钟的再生成窗口内对公众可见，或在手动触发再验证后 1 分钟内可见。
- **SC-005**: Dashboard 中展示的文章数量和总阅读量，在内容更新后 1 分钟内应与实际数据保持一致。
- **SC-006**: 在 375px 移动端宽度和 1440px 桌面端宽度下，首页、文章详情页、文章编辑页和后台列表页均不得出现影响核心操作的布局错乱或横向滚动。

## Assumptions（假设）

- 本功能面向单站点的个人博客系统，不包含多租户或多站点管理能力。
- 评论在 v1 中为单层评论，不包含评论回复、点赞或举报流程。
- 邮箱注册登录在 v1 中不强制包含邮箱验证、找回密码或复杂账户安全流程。
- 公开访客只能访问 `published` 文章；`draft` 和 `archived` 默认不对公众可见。
- About 页面内容在 v1 中按静态页面维护，不纳入后台 CMS 编辑范围。
- 当文章未显式配置 SEO 字段时，系统可以使用文章标题、摘要和封面图作为默认回退值。
- 阅读量统计允许采用去重或限流策略，以降低短时间刷新带来的异常放大。
- GitHub OAuth 登录依赖预先配置好的 GitHub 应用凭据。
- 图片上传在 v1 中使用本地存储实现，但必须抽象出可替换的存储接口，以便后续接入云存储。
